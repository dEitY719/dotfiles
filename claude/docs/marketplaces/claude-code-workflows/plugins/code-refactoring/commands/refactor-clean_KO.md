## 📋 개요

사용자가 코드를 더 깔끔하고 유지보수하기 쉽게 만들기 위해 리팩토링 도움이 필요할 때 사용됩니다. 과도한 엔지니어링 없이 코드 품질을 향상시키는 실용적인 개선에 중점을 둡니다.

## 1️⃣ 코드 분석

현재 코드를 다음 항목에 대해 분석합니다:

### 🔴 코드 냄새(Code Smells)
- 긴 메서드/함수 (20줄 이상)
- 큰 클래스 (200줄 이상)
- 중복 코드 블록
- 죽은 코드 및 사용하지 않는 변수
- 복잡한 조건문 및 중첩 루프
- 매직 넘버 및 하드코딩된 값
- 불명확한 이름 규칙
- 컴포넌트 간의 강한 결합
- 누락된 추상화

### 🔴 SOLID 원칙 위반
- **SRP(단일 책임 원칙)** 위반
- **OCP(개방-폐쇄 원칙)** 문제
- **LSP(리스코프 치환 원칙)** 문제
- **ISP(인터페이스 분리 원칙)** 우려사항
- **DIP(의존성 역전 원칙)** 위반

### 🔴 성능 문제
- 비효율적인 알고리즘 (O(n²) 이상)
- 불필요한 객체 생성
- 메모리 누수 가능성
- 블로킹 작업
- 캐싱 기회 부재

## 2️⃣ 리팩토링 전략

우선순위가 정해진 리팩토링 계획을 수립합니다:

### ⚡ 즉각적인 수정 (높은 영향도, 낮은 노력)
- 매직 넘버를 상수로 추출
- 변수 및 함수 이름 개선
- 죽은 코드 제거
- 부울 표현식 단순화
- 중복 코드를 함수로 추출

### 🔧 메서드 추출
- 긴 메서드를 여러 개의 작은 메서드로 분해
- 각 메서드는 단일 책임 수행

### 📦 클래스 분해
- 책임을 별도 클래스로 추출
- 의존성을 위한 인터페이스 생성
- 의존성 주입 구현
- 상속보다 합성 사용

### 🎨 패턴 적용
- **Factory 패턴**: 객체 생성
- **Strategy 패턴**: 알고리즘 변형
- **Observer 패턴**: 이벤트 처리
- **Repository 패턴**: 데이터 접근
- **Decorator 패턴**: 동작 확장

## 3️⃣ 실제 SOLID 원칙 적용

### 📍 단일 책임 원칙(SRP)
각 클래스는 변경 이유가 하나만 있어야 합니다.

**문제점**: 한 클래스에 데이터 검증, 저장, 이메일 발송, 로깅이 섞여 있음
**해결책**: 각 책임을 별도 클래스로 분리

### 📍 개방-폐쇄 원칙(OCP)
확장에는 개방적이고 수정에는 폐쇄적이어야 합니다.

**문제점**: 새로운 할인 유형 추가 시마다 코드 수정 필요
**해결책**: Strategy 패턴으로 다양한 할인 전략 구현

### 📍 리스코프 치환 원칙(LSP)
자식 클래스는 부모 클래스 자리에 완벽히 대체 가능해야 합니다.

**문제점**: Square 클래스가 Rectangle의 동작을 변경
**해결책**: 공통 인터페이스로 독립적 구현

### 📍 인터페이스 분리 원칙(ISP)
불필요한 메서드 구현을 강요하는 큼직한 인터페이스 제거

**문제점**: Robot이 `eat()`, `sleep()` 메서드를 불필요하게 구현
**해결책**: 작은 특화된 인터페이스로 분리

### 📍 의존성 역전 원칙(DIP)
고수준 모듈이 저수준 모듈에 의존하지 않음. 추상화에 의존

**문제점**: UserService가 구체적인 MySQLDatabase 클래스에 의존
**해결책**: Database 인터페이스를 통한 느슨한 결합

**리팩토링 후**:
- 엔티티 도메인 계층
- 저장소 계층 (데이터 접근)
- 유효성 검사 계층
- 애플리케이션 서비스 계층
- 이메일 서비스 계층

### 🐛 시나리오 2: 코드 냄새 해결 카탈로그

| 냄새 | 문제 | 해결책 |
|------|------|--------|
| 긴 매개변수 목록 | 가독성 저하 | Parameter Object 패턴 사용 |
| Feature Envy | 다른 클래스 데이터 과다 사용 | 메서드를 해당 클래스로 이동 |
| Primitive Obsession | 기본형 타입 남용 | Value Object 패턴 적용 |
| Switch 문 남용 | 새 타입 추가 시 수정 필요 | Strategy/Polymorphism 사용 |

## 5️⃣ 의사결정 프레임워크

### 📊 코드 품질 메트릭 해석 매트릭스

| 메트릭 | 양호 | 주의 | 심각 | 조치 |
|--------|------|------|------|------|
| 순환 복잡도 | <10 | 10-15 | >15 | 메서드 분할 |
| 메서드 줄 수 | <20 | 20-50 | >50 | 메서드 추출 |
| 클래스 줄 수 | <200 | 200-500 | >500 | 클래스 분해 |
| 테스트 커버리지 | >80% | 60-80% | <60% | 테스트 추가 |
| 코드 중복 | <3% | 3-5% | >5% | 공통 코드 추출 |

### 📈 리팩토링 ROI 분석

```
우선순위 = (비즈니스 가치 × 기술 부채) / (노력 × 위험도)
```

**비즈니스 가치 (1-10)**:
- 핵심 경로 코드: 10
- 자주 변경됨: 8
- 사용자 대면 기능: 7
- 기술 부채만 높음: 2

**노력 (시간)**:
- 변수명 변경: 1-2
- 메서드 추출: 2-4
- 클래스 리팩토링: 4-8
- 아키텍처 변경: 40+

### 🌳 기술 부채 우선순위 결정 트리

```
프로덕션 버그 발생?
├─ YES → 우선순위: 긴급 (즉시 수정)
└─ NO → 새 기능 개발 차단?
    ├─ YES → 우선순위: 높음 (이번 스프린트)
    └─ NO → 자주 수정되는 코드?
        ├─ YES → 우선순위: 중간 (다음 분기)
        └─ NO → 테스트 커버리지 < 60%?
            ├─ YES → 우선순위: 중간 (테스트 추가)
            └─ NO → 우선순위: 낮음 (백로그)
```

## 7️⃣ 리팩토링된 구현

완전히 리팩토링된 코드는 다음 사항을 포함합니다:

### ✨ 클린 코드 원칙
- **의미 있는 이름**: 검색 가능하고 발음하기 쉬운 이름
- **단일 책임**: 함수는 한 가지만 잘함
- **부작용 없음**: 예측 가능한 동작
- **일관된 추상화**: 같은 수준의 개념 혼합 금지
- **DRY**: 중복 제거 (Don't Repeat Yourself)
- **YAGNI**: 필요하지 않은 기능 추가 금지

### 🚨 에러 처리
- 구체적인 예외 클래스 사용
- 명확한 에러 메시지
- 빠른 실패(Fail Fast) 원칙

### 📚 문서화
- 함수별 목적, 매개변수, 반환값 설명
- 복잡한 로직에 주석 추가
- 변경 사항 이력 기록
