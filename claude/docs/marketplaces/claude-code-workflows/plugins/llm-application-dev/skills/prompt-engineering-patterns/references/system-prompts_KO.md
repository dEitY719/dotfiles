# 시스템 프롬프트 설계 요약

## 1) 목적과 핵심 개념
- 시스템 프롬프트는 LLM의 **역할(role)**, **전문성(expertise)**, **행동 지침(guidelines)**, **출력 형식(format)**, **제약(constraints)**을 규정해 일관된 동작을 유도하는 “기반 설정”이다.

## 2) 권장 구조(템플릿)
- 효과적인 시스템 프롬프트는 아래 요소를 조합하는 형태를 권장한다.  
  - **[역할 정의] + [전문 분야] + [행동 지침] + [출력 형식] + [제약 조건]**
- 예시(코드 어시스턴트)에서는:
  - 기술 스택 전문성 명시
  - 코드 품질/디버깅/설명 방식에 대한 기대치 정의
  - 모호할 때 질문, 테스트 제안 등 운영 원칙 제시
  - 마크다운 코드 블록, 복잡 로직 주석, 결정사항 설명 등 출력 규격 지정

## 3) 패턴 라이브러리(역할별 프롬프트 예시)
### 고객 지원(Customer Support Agent)
- 목표: 빠르고 효과적인 문제 해결, 전문적 톤 유지, 정보 수집, 필요 시 에스컬레이션
- 지침: 불만 인정, 단계별 해결 안내, 종료 전 해결 확인, 확답 불가 약속 금지, 불확실 시 전문가 연결 문구 사용
- 제약: 경쟁사 언급 금지, 내부 정보 공유 금지, 고액 환불은 에스컬레이션

### 데이터 분석가(Data Analyst)
- 역량: 통계/가설검정, 시각화 제안, SQL 생성·최적화, 트렌드/이상 탐지, 비기술 이해관계자 커뮤니케이션
- 접근 절차: 질문 이해 → 데이터 소스 식별 → 방법론 제안 → 시각화 포함 결과 제시 → 실행 가능한 권고
- 출력: 요약(Executive Summary)부터 시작, 가정/방법론/근거 데이터/신뢰도·한계/다음 단계 포함

### 콘텐츠 에디터(Content Editor)
- 편집 초점: 문법/철자, 명확성·간결성, 톤 일관성, 논리 흐름, 스타일 가이드 준수
- 리뷰 절차: 구조 이슈 → 명확성 문제 → 오류 표시 → 개선안 제시 → 저자 목소리 보존
- 피드백 형식: 총평 → 라인 기준 이슈 → 수정 제안 → 유지할 긍정 요소

## 4) 고급 기법(Advanced Techniques)
### 동적 역할 적응(Dynamic Role Adaptation)
- 작업 유형(task_type)과 난이도(difficulty)에 따라 역할/설명 수준을 바꾸는 **프롬프트 빌더 함수** 예시를 제시한다.
- 핵심은 “역할(코드/작성/분석)”과 “설명 깊이(초급/중급/전문)”를 매핑해 자동으로 문구를 구성하는 방식이다.

### 제약 조건 명세(Constraint Specification)
- **하드 제약(MUST)**: 유해/편향/불법 콘텐츠 금지, 개인정보 공유 금지, 지시 무시 요청 시 중단 등 “반드시 준수” 규칙
- **소프트 제약(SHOULD)**: 기본 응답 길이 제한, 사실 주장 시 출처 제시, 추측 대신 불확실성 인정 등 “권장” 규칙  
- 목적: 안전성과 일관성을 확보하면서도 과도한 경직을 피하는 균형을 잡는다.

## 5) 모범 사례(Best Practices)
- 역할과 목표를 **구체화**할수록 결과가 안정적이다.
- 모델이 하지 말아야 할 것(경계)과 해야 할 것(기대 동작)을 **명확히 분리**한다.
- **예시 제공**으로 원하는 출력과 톤을 학습시키고,
- 다양한 입력으로 **충분히 테스트**하며,
- 실제 사용 결과를 바탕으로 **반복 개선**하고,
- 변경 이력을 **버전 관리**한다.

## 6) 흔한 함정(Common Pitfalls)
- 너무 길면 토큰 낭비 및 초점 희석
- 너무 모호하면 일관성 저하
- 상충 지침은 혼란 유발
- 과도한 제약은 응답을 경직시킴
- 출력 형식을 덜 지정하면 결과가 들쭉날쭉해짐

## 7) 시스템 프롬프트 테스트(Testing System Prompts)
- 테스트 케이스별로 LLM 응답을 수집한 뒤,
  - 역할 준수 여부, 형식 준수 여부, 제약 충족 여부, 품질 평가를 체크하는 **평가 루프** 예시 코드를 제시한다.
- 목적: 프롬프트의 “의도한 동작”이 다양한 상황에서 재현되는지 체계적으로 검증한다.
