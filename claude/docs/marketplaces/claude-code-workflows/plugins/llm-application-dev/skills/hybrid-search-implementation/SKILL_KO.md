# Hybrid Search Implementation (요약)

벡터 유사도 검색(semantic)과 키워드 기반 검색(BM25/FTS 등, exact)을 **동시에 수행**한 뒤, 결과를 **융합(fusion)** 또는 **재랭킹(rerank)** 하여 더 좋은 검색 품질을 얻는 구현 패턴이다.

## 언제 사용하나

- RAG에서 **재현율(recall)** 을 높이고 싶을 때
- 의미 기반 검색의 장점(유사 의미)을 유지하면서 **정확 일치(이름/코드/고유명사)** 도 놓치지 않게 하고 싶을 때
- 도메인 특화 용어가 많아 **벡터 검색만으로 누락**이 발생할 때
- 벡터/키워드 중 한 방식만으로는 성능이 부족할 때

## 핵심 개념

### 1) 하이브리드 아키텍처

- 입력 쿼리를 기준으로 **벡터 검색**과 **키워드 검색**을 병렬(또는 순차)로 수행
- 각 검색이 반환한 후보(candidates)를 **결합(Fusion)** 하여 최종 순위를 만든다

### 2) 결과 결합(Fusion) 방식

- `RRF(Reciprocal Rank Fusion)`: 각 리스트의 **순위(rank)** 를 기반으로 점수를 합산하는 범용 방식(튜닝 부담이 적음)
- `Linear(선형 결합)`: 점수들을 정규화한 뒤 가중합으로 결합(가중치 `alpha` 등 튜닝 가능)
- `Cross-encoder 재랭킹`: 후보를 모아 **신경망 모델로 재정렬**(품질 최고 수준, 비용/지연 증가)
- `Cascade(캐스케이드)`: 먼저 필터/경량 랭킹 → 이후 재랭킹으로 효율 최적화

## 제공 템플릿(무엇이 들어있나)

### Template 1: RRF + 선형 결합(Python)

- 여러 랭킹 리스트를 `RRF`로 합치는 함수와,
- 벡터 점수/키워드 점수를 `[0,1]`로 정규화 후 `alpha`로 섞는 선형 결합 예시를 제공한다.

### Template 2: PostgreSQL 하이브리드(pgvector + FTS)

- `pgvector` 확장과 HNSW 벡터 인덱스, `tsvector` 기반 GIN FTS 인덱스를 함께 구성
- 벡터 검색과 FTS 검색을 CTE로 뽑고, `FULL OUTER JOIN`으로 합친 뒤 `RRF` 점수로 정렬
- 옵션으로 메타데이터 필터링을 적용하며, 추가로 `CrossEncoder`를 이용한 재랭킹 흐름도 포함한다.

### Template 3: Elasticsearch 하이브리드(dense_vector + BM25)

- `dense_vector` 필드(코사인 유사도)와 `text` 필드를 함께 사용
- 한 쿼리에서 벡터 기반 `script_score`와 텍스트 `match(BM25)`를 `should`로 결합하는 방식
- Elasticsearch 8.x의 `RRF` 랭킹 기능을 활용하는 별도 예시도 제공한다.

### Template 4: 커스텀 Hybrid RAG 파이프라인(전체 흐름)

- 임베딩 생성 → 벡터/키워드 검색 병렬 실행 → `RRF` 또는 선형 융합 → (선택) 재랭킹 → `top_k` 반환
- 결과 객체에 `source`(vector/keyword/hybrid) 및 `metadata`를 포함해 추적 가능성을 높이는 구조를 제시한다.

## 베스트 프랙티스

- 가중치/파라미터는 **실데이터로 실험적으로 튜닝**
- 간단히 시작하려면 `RRF`가 유리(튜닝 최소)
- 품질이 중요하면 **재랭킹 추가**(대신 비용/지연 고려)
- 벡터/키워드 점수를 **둘 다 로깅**해 디버깅 용이하게
- A/B 테스트로 실제 사용자 지표 개선을 검증

## 주의사항(안티패턴)

- 모든 쿼리에 동일 설정이 최적이라고 가정하지 말 것
- 키워드 검색을 생략하지 말 것(정확 일치에 강함)
- 후보를 과도하게 많이 가져오지 말 것(재현율-지연 트레이드오프)
- 빈 결과/단어 1개 쿼리 등 엣지 케이스를 무시하지 말 것
