# Service Mesh Expert 요약

## 개요
- Kubernetes 환경에서 마이크로서비스 간 통신을 **서비스 메시(Istio/Linkerd)**로 표준화하고, **제로 트러스트 네트워킹**, **트래픽 제어**, **관측성**, **멀티 클러스터 구성**을 설계·구현·최적화하는 역할의 에이전트/스킬입니다.

## 주요 기능 (Capabilities)
- **Istio/Linkerd 설치·구성·최적화**
- **트래픽 관리**: 라우팅, 로드 밸런싱, 서킷 브레이킹, 재시도(retry)
- **mTLS 구성 및 인증서 관리**
- **관측성 통합**: 분산 추적(Distributed Tracing) 기반 메시 관측성 구성
- **멀티 클러스터/멀티 클라우드** 메시 연합(federation) 설계
- **점진적 배포(Progressive Delivery)**: 카나리, 블루-그린 배포 전략
- **보안 정책**: 보안 정책 및 인가(Authorization) 규칙 설계·적용

## 사용 시점 (When to Use)
- Kubernetes에서 **서비스 간 통신**을 메시로 구현/정비할 때
- **mTLS 기반 제로 트러스트**를 도입하거나 강화할 때
- **카나리 트래픽 분할** 등 트래픽 스플리팅이 필요할 때
- **서비스 메시 연결/통신 장애**(connectivity) 디버깅이 필요할 때
- **레이트 리미팅, 서킷 브레이커** 등 회복탄력성 패턴을 적용할 때
- **클러스터 간 서비스 디스커버리**를 구성할 때

## 권장 워크플로 (Workflow)
1. 현 인프라/요구사항 평가
2. 메시 토폴로지 및 트래픽 정책 설계
3. 보안 정책 구현(mTLS, AuthorizationPolicy)
4. 관측성 구성(메트릭, 트레이스, 로그)
5. 트래픽 관리 규칙 설정
6. 장애 조치(failover) 및 회복탄력성 패턴 테스트
7. 운영 런북(runbook) 문서화

## 모범 사례 (Best Practices)
- 초기에는 **permissive 모드**로 시작하고, 단계적으로 **strict mTLS**로 강화
- **네임스페이스**를 활용해 정책 격리(isolation) 수행
- 필요해지기 전에 **서킷 브레이커**를 선제 도입
- 메시 오버헤드(**지연, 리소스 사용량**) 지속 모니터링
- 사이드카 리소스 크기(요청/제한)를 적절히 산정
- 일관된 로드 밸런싱을 위해 **DestinationRule** 활용
