# Release Notes 마크다운 템플릿

프로젝트에 이전 릴리즈 노트가 있다면 **그 포맷을 따르는 것이 최우선**. 없으면 아래 템플릿을 기본으로 사용하고 필요한 섹션만 남긴다.

## 표준 템플릿

```markdown
# <Project> <version> Release Notes

**릴리즈 날짜**: YYYY-MM-DD
**커밋 수**: N
**개발 기간**: YYYY-MM-DD ~ YYYY-MM-DD
**이전 버전**: <prev-version>

---

## 개요

<prev-version> 이후 N번째 마일스톤 릴리즈입니다.
<핵심 변경사항을 1~2문장으로 요약 — 커밋 나열 금지>

---

## 주요 변경사항

### 1. <테마 제목 — 사용자 기능명으로>

- **<기능명>**: <사용자 관점 변화 설명>
- **<기능명>**: <사용자 관점 변화 설명>

### 2. <테마 제목>

- **<기능명>**: <설명>

---

## 버그 수정

- <기능과 무관한 standalone fix 항목들>
- <항목>

---

## 리팩토링

- <standalone refactor 항목들 — 사용자 영향이 있는 것만>

---

## 문서

- <docs 항목들>

---

## 📢 사용자 공지용 요약 (선택)

Slack / Teams / 이메일 등 외부 공지 채널에 복붙할 수 있는 짧은 버전.

**<version> 주요 변경**

- 🎨 <사용자 관점 기능 설명>
- ⚡ <성능 / UX 개선>
- 🐛 <중요 버그 수정>

**🛠️ 내부 개선**

- 🔧 <리팩토링 / 인프라>
- 📚 <문서 / 가이드>
```

## 작성 팁

### 개요
- 커밋 나열 ❌
- 이 릴리즈의 **성격** (기능 추가 중심? 안정화? 리팩토링?) 을 한 문장
- 그 다음 핵심 변화 1-2개를 한 문장

**좋은 예**:
> v0.2.0 이후 5번째 마일스톤 릴리즈입니다. 그래프 뷰의 인터랙션이 전면 재설계되었고, 검색 파이프라인이 한/영 혼용 입력을 자동으로 처리합니다.

**나쁜 예**:
> feat: 호버 카드 추가, feat: 패닝 개선, feat: 검색 IME 처리... (커밋 나열)

### 메타데이터
- `개발 기간`: 앵커 커밋 날짜 ~ HEAD 날짜
- `커밋 수`: `git log --oneline <anchor>..HEAD | wc -l`
- `이전 버전`: 태그명 or 릴리즈 노트 파일명에서 추출

### 공지용 요약 (📢 섹션)
- **선택**이지만 팀 공지 문화가 있다면 강력 추천
- 이모지는 카테고리 시각화용 (🎨 UI, ⚡ 성능, 🐛 버그, 🔧 인프라, 📚 문서, 🔒 보안)
- 3~7개 bullet이 적절. 10개 넘으면 읽히지 않음

### 항목 설명
- 주어는 암묵적으로 "사용자는" / "이제"
- "무엇을 할 수 있게 됐나" / "무엇이 달라졌나" 관점
- 버전별 breaking change는 **별도 🚨 섹션**으로 분리 권장

## 저장 위치

프로젝트 관례 순서대로 확인:

1. `docs/release-notes/<version>.md` (가장 흔한 패턴)
2. `CHANGELOG.md` (하단에 append)
3. `RELEASES.md` 또는 `docs/CHANGELOG.md`
4. 없으면 `docs/release-notes/<version>.md`로 새로 시작

파일명 포맷도 이전 릴리즈 파일을 따른다 (예: `v0.2.0.md` vs `0.2.0.md` vs `2024-04-15-v0.2.0.md`).
