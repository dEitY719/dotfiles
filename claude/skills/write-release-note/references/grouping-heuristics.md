# 커밋 그룹핑 휴리스틱

## 기본 원칙

이 단계는 **창의적 판단**이 필요하다. 커밋을 1:1로 나열하지 말고, 관련 커밋들을 사용자 관점 테마로 묶는다.

## 그룹핑 규칙

| 패턴 | 어디에 묶을지 |
|------|---------------|
| 같은 컴포넌트의 여러 `feat:` | 하나의 테마 섹션 |
| 같은 영역의 `feat:` + 여러 `fix:` + `refactor:` | 하나의 섹션 (기능 개발 전체 사이클) |
| 기능과 무관한 독립 `fix:` | **"버그 수정"** 섹션 |
| 독립 `refactor:` | **"리팩토링"** 섹션 |
| `docs:` | **"문서"** 섹션 |
| `chore:`, `test:`, `ci:` | 일반적으로 생략. 중요한 경우(의존성 메이저 업그레이드 등)만 언급 |

## 테마 네이밍 규칙

**사용자가 보는 기능 이름**을 쓴다. 구현 디테일은 절대 금지.

| ✓ 좋은 예 | ✗ 나쁜 예 |
|-----------|-----------|
| 그래프 뷰 호버 카드 재설계 | react-force-graph-2d canvas overlay refactor |
| 검색 결과 정렬 개선 | lodash.orderBy multi-key comparator |
| 인증 플로우 간소화 | OAuth2 middleware refactor |
| 다크 모드 지원 추가 | Tailwind dark: variant 전면 적용 |
| CSV 내보내기 속도 개선 | streaming writer로 전환 |

## 판단 기준

섹션 제목을 썼을 때 다음을 자문한다:

> "사용자가 이 제목을 보고 *무엇이* 달라졌는지 바로 이해할 수 있는가?"

- **Yes** → 좋은 테마
- **No** (내부 기술 용어가 필요함) → 구현 용어일 가능성, 재작성

## 기능 관련 수정의 배치

버그 수정이지만 **그 릴리즈의 특정 기능과 연결**되어 있으면 해당 기능 테마 섹션 안에 넣는다.

**예시**:
- 이번 릴리즈에서 "그래프 뷰 호버 카드"를 새로 만들었고
- `fix: 호버 카드 깜빡임 수정` 커밋이 있다면
- → "버그 수정" 섹션이 아니라 "그래프 뷰 호버 카드 재설계" 섹션 안에 배치

이렇게 해야 사용자가 "이 기능은 아직 불안정하구나"가 아니라 "이 기능이 완성도 있게 들어왔구나"로 받아들인다.

## 항목 설명 작성법

각 bullet은 "무엇을 했다"가 아니라 "사용자에게 어떤 변화가 있다"로 쓴다.

| ✓ | ✗ |
|---|---|
| 큰 노드 집합에서도 끊김 없이 패닝 가능 | `useCallback`으로 핸들러 메모이제이션 |
| 다크 모드에서 가독성 향상 | 색상 토큰 대비비 재계산 |
| 검색 시 한/영 자동 전환 지원 | IME composition 이벤트 처리 추가 |

## 순서 정하기

"주요 변경사항" 섹션 내 테마 순서는 **사용자 영향이 큰 순서**로:
1. 새로운 기능 / 워크플로 변경 (가장 눈에 띄는 변화)
2. 기존 기능 개선
3. 성능 / UX 개선
4. 내부 개선 (optional 섹션으로 이동 가능)
