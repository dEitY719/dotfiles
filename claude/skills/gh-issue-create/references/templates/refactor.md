# refactor — 본문 템플릿

동작 보존을 전제로 한 구조 변경 이슈에 사용한다.

## 타이틀

```
refactor[(<scope>)]: <한 줄 요약>
```

예) `refactor(shell-common): gcp_scan 의 subshell 트레이싱 패턴 정리`

## 본문 골격

```markdown
## TL;DR
<1~3줄 — 무엇을 어떻게 바꾸고 왜 지금인가>

## 동기 (Why now)
<현재 구조의 고통점·미래 작업 차단 요인. 기능 추가 의도가 섞이면
별도 feat 이슈로 분리>

## 범위 (Scope)
<수정 대상 파일·모듈. "여기는 안 건드린다" 도 명시>

## Before / After
<핵심 변경 1~3건만. diff 또는 구조 다이어그램>

## 동작 보존 보장 (Behavior Preservation)
<기존 테스트·스냅샷·계약. 회귀 검출 방법>

## 리스크·롤백 계획
<공유 코드·외부 의존 영향. 단계별 롤백 트리거>

## 검증
<통과해야 할 테스트·수동 시나리오>

## References
- 관련 파일: `path:line`
- 관련 이슈/PR/커밋:
```

## 작성 노트

- 동작 변경이 섞이면 refactor 가 아니라 feat / fix 다. 본 템플릿은
  순수 구조 변경에만 사용.
- "리스크·롤백 계획" 은 한 줄이라도 반드시 채운다 — 동작 보존
  주장이 무너졌을 때를 대비.
