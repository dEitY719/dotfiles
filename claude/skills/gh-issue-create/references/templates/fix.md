# fix — 본문 템플릿

버그 수정 이슈에 사용한다. 기존 `bug` 카테고리를 흡수.

## 타이틀

```
fix[(<scope>)]: <증상 한 줄 요약>
```

예) `fix(project-board): --only-from 가드가 In review 교정을 막음`

## 본문 골격

```markdown
## TL;DR
<1~3줄 — 무엇이 어떤 환경에서 실패하는가>

## 증상 (Symptom)
<에러 메시지·실패 모드 — 사용자에게 보이는 형태>

## 재현 (Reproduction)
<명령·환경·전제 — 차후 fix 검증 시 그대로 돌릴 수 있게>

## 근본 원인 (Root Cause)
<원인 추적 과정과 결론. 코드 경로 `path:line` 인용 권장>

## 수정안 (Fix Plan / Solution)
<수정 방향. 코드 diff 또는 의사 코드. 대안이 있으면 비교 표>

## 영향 범위 (Impact)
<누가/언제 영향받는가, 회피책, 우선순위>

## 검증 (Verification)
<수정 후 통과해야 할 테스트·시나리오>

## 참고 (References)
- 관련 파일: `path:line`
- 관련 이슈/PR/커밋:
- 환경: (OS, 버전 등)
```

## 작성 노트

- "재현" 과 "검증" 은 별도 섹션으로 둔다 — 재현은 fix 전 상태,
  검증은 fix 후 통과 기준.
- "근본 원인" 은 결론만이 아니라 추적 과정도 포함한다 — 후속 회귀
  방지·블로그 재활용 시 가치가 큼.
- 부수 버그가 있으면 별도 이슈로 분리하되, 본 이슈 References 에서
  교차 링크.
