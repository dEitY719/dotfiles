# Clarification & Scope Guard

Step 2 와 Step 3 사이에서 호출되는 "이슈 본문 작성 직전" 안전망. 모호한
요청 / 과대 스코프를 1~2줄 확인으로 잡아낸다. 사용자가 명시적으로
"한 이슈로" 라고 답하면 그대로 진행한다 — 강제로 분리하지 않는다.

## 트리거 신호 (하나 이상 매치 시 확인)

1. **동사 없는 명사 나열** — 예: "사용자 관리 기능", "결제 시스템"
   → 무엇을 추가/변경/제거하려는지 1줄 확인.
2. **컴포넌트 3개 이상 혼재** — 한 요청에 백엔드/프론트/CLI/스킬 등
   3개 이상 영역이 동시에 등장 → 이슈 분리 제안.
3. **feature 범위 미정의** — 대화가 "X 어떻게?" "Y 도 필요" 식으로
   끝나서 acceptance criteria 가 도출 안 되는 상태 → 1줄 확인.

위 셋 중 어느 것도 매치 안 되면 본 가이드는 no-op. 곧장 Step 3 로 진행.

## 스코프 과대 분리 제안 — 형식

여러 독립 기능이 한 요청에 들어 있다면, 묻기 전 분리안을 제시:

```
이것은 여러 이슈로 분리하면 좋을 것 같습니다:
  - feat(auth): 로그인 페이지
  - feat(profile): 프로필 수정
단일 이슈로 생성하시겠습니까? (y/n)
```

응답 처리:

- `y` / "한 이슈" / 묵시적 동의 → 그대로 한 이슈로 생성. 이때 본문
  Goals 에 분리안을 보존해 둔다 ("이후 PR 단위로 쪼갤 수 있음").
- `n` / "분리" → 가장 큰 범위 한 건만 이번에 만들고, 나머지는 사용자가
  /gh-issue-create 를 다시 호출하도록 안내. 자동 다중 생성 금지.

## 모호함 해소 — 형식

명사 나열·범위 미정의 케이스는 길게 묻지 말고 1~2줄로:

```
이슈를 생성하기 전에 한 가지만 확인:
- 이것은 <X> 를 <추가/수정/제거> 하는 작업으로 정리하면 될까요?
```

응답이 단답("응" / "yes")이라도 받으면 본문에 그대로 사용. 응답이
오기 전에는 절대 `gh issue create` 를 호출하지 않는다.

## 적용 범위

이 가이드는 `feat` / `refactor` / `chore` 처럼 스코프가 커질 수 있는
prefix 에 가장 자주 매치된다. `fix` / `docs` / `test` 는 보통 트리거가
발화하지 않으므로 no-op 된다.

## 비목적

- REQ-ID 시스템 (`REQ-B-Access-1` 등) 도입하지 않음 — 프로젝트 특화
  포맷이라 범용 이슈 생성과 미스매치.
- multi-agent peer review (v1/v2) 도입하지 않음 — `gh:issue-create` 의
  단일-호출 계약과 충돌.
