# Doc Co-Authoring Skill

**구조화된 문서 작성 워크플로우** - 기술 스펙부터 의사결정 문서까지, 체계적으로 검증된 문서를 함께 만듭니다.

---

## 🎯 이 스킬이 뭔가요?

복잡한 기술 문서나 비즈니스 제안서를 혼자 작성하기 어렵나요? 이 스킬은 **3단계 검증 워크플로우**로 당신과 함께 문서를 만들어갑니다.

마치 편집자가 작가의 글을 다듬는 것처럼, 우리는 컨텍스트를 수집하고 섹션별로 다듬으며, 최종적으로 **독자 관점에서 검증**합니다.

---

## 🔄 어떻게 작동하나요?

### 3단계 프로세스

```text
1. 컨텍스트 수집 → 2. 문서 작성 및 다듬기 → 3. 독자 테스트
```text

#### Stage 1: Context Gathering (컨텍스트 수집)

**목표**: 당신이 알고 있는 모든 것을 Claude가 알도록 하기

- 메타정보: 문서 타입, 대상 독자, 목표

- 정보 수집: 배경, 팀 논의, 기술 사항, 정치적 맥락

- 명확화: 5-10개 질문으로 불분명한 부분 확인

**시간**: 5-15분 (정보 수집 방식에 따라)

#### Stage 2: Refinement & Structure (섹션별 작성)

**목표**: 섹션별로 브레인스토밍 → 선별 → 작성 → 다듬기

각 섹션마다:
1. **명확화**: 이 섹션에 뭘 포함할지?
2. **브레인스토밍**: 5-20개 후보 항목 생성
3. **선별**: 어떤 항목을 유지할지 결정
4. **작성**: 실제 텍스트 작성
5. **다듬기**: 반복해서 수정

**시간**: 섹션당 10-20분

#### Stage 3: Reader Testing (독자 검증)

**목표**: 실제 독자 입장에서 문서가 작동하는지 확인

- 예상 질문 생성: 독자가 할 법한 5-10개 질문

- 새로운 Claude로 테스트: 컨텍스트 없이 문서만 주고 질문 답변

- 문제 식별: 불명확함, 가정, 모순 찾기

- 수정: 문제 섹션 다시 작성

**시간**: 10-15분

---

## 📊 Output (생산물)

### 최종 산출물

완전히 검증된, 독자가 이해할 수 있는 문서:

```text
┌────────────────────────────────┐
│  1. Executive Summary           │
│  2. Problem & Context           │
│  3. Proposed Solution           │
│  4. Implementation Plan         │
│  5. Risk & Mitigation           │
│  6. Appendix (if needed)        │
└────────────────────────────────┘
```

**특징**:

- 완전한 컨텍스트 반영

- 일관된 톤과 깊이

- Reader tested - 실제 독자가 이해 가능

- 편집됨 - 불필요한 부분 제거

## 💡 실제 사용 예시

### 요청

> "기술 스펙 문서를 작성해야 해. 새 마이크로서비스 아키텍처에 대한 거야."

### Stage 1: Context Gathering

```text
Claude: 몇 가지 물어보겠습니다.
1. 주요 독자는 누구? (엔지니어, PM, 의사결정자?)
2. 현재 아키텍처의 문제점은?
3. 새 아키텍처의 핵심 개선사항은?
[...]

You: 독자는 엔지니어들이고...
```

### Stage 2: Refinement & Structure

```text
Claude: 이 섹션들이 필요할 것 같습니다:

- Current Architecture

- Proposed Changes

- Migration Plan

- Performance Impact

어떤 순서로 할까요?
```

각 섹션을 하나씩 다듬으면서 실제 문서 완성.

### Stage 3: Reader Testing

```text
Claude: 새로운 엔지니어가 이 문서를 읽고 물어볼 것 같은 질문들:
1. "기존 시스템은 언제까지 유지되나요?"
2. "마이그레이션 중에 다운타임이 있나요?"
[...]

각각 테스트해봤고, 섹션 3 수정이 필요해 보입니다.
```

## 🎯 특징

| 특징 | 설명 |
| --- | --- |

| **구조화된 접근** | 3단계로 체계적 진행 |
| **정보 손실 최소화** | 컨텍스트 수집으로 모든 정보 활용 |
| **반복적 개선** | 브레인스토밍 → 선별 → 작성 → 다듬기 |
| **Reader-focused** | 최종적으로 실제 독자 기준으로 검증 |
| **유연한 속도** | 빨리 진행하거나 천천히 할 수 있음 |
| **일관된 톤** | 전체 문서 톤과 깊이 일관성 |

## 🚀 시작하기

### 사용자 입장 (Claude 사용할 때)

```text
"기술 스펙 문서를 작성해야 해."
```text

또는

```text
"의사결정 문서를 함께 만들어줄래? 구조화된 워크플로우로."
```

### 예상 시간

- **전체**: 30-90분 (문서 복잡도에 따라)

- **각 Stage**: 10-20분

- **수정**: 필요한만큼 (보통 1-3회 반복)

### 문서 타입

지원하는 모든 문서:

- 기술 스펙 (Technical Specification)

- 의사결정 문서 (Decision Doc)

- PRD (Product Requirements Document)

- 제안서 (Proposal)

- RFC (Request for Comments)

- 프로젝트 계획서

- 리스크 분석

- 등등

## 🛠️ 기술 스택

- **문서 형식**: Markdown 또는 공유 문서 (Google Docs, Notion 등)

- **컨텍스트 통합**: MCP 기반 (Slack, Google Drive 등에서 정보 자동 수집)

- **검증**: 별도 Claude 인스턴스로 Reader testing

---

## 📚 스킬의 핵심 철학

> **"좋은 문서는 공동 작업의 산물이다"**

- 혼자 쓰기 보다는 함께 만들기

- 컨텍스트가 완전해야 좋은 문서가 나옴

- 독자 관점에서 검증이 필수

- 반복은 품질의 친구

## ❓ FAQ

**Q: 정말 Reader testing이 중요한가요?**

A: 매우 중요합니다! 작가 입장에서 당연한 것이 독자에겐 불명확할 수 있거든요.

**Q: 템플릿을 제공하나요?**

A: 네! 각 문서 타입별 템플릿이 있습니다. 필요하면 로드할 수 있습니다.

**Q: 이미 작성한 문서를 개선할 수 있나요?**

A: 물론입니다! 기존 문서를 업로드하면 3단계 프로세스로 개선합니다.

**Q: 워크플로우를 스킵할 수 있나요?**

A: 네, 하지만 권장하지 않습니다. 각 단계가 최종 품질을 보장합니다.
