# 운영 방식 (Micro-control 스타일)

## 0) 원칙: 작업 단위를 “티켓/주제”로 분리

- Jira 단위(티켓): 배포/릴리즈/버그/개선 등 “추적”이 필요한 일
- Confluence 단위(팁): 재사용 가능한 “지식” (원인/해결/재현/주의사항)
- Git 단위(커밋/PR): Jira 티켓 키를 커밋 메시지/브랜치에 포함 (가능하면)

## 1) 외부 PC에서의 기록 루틴 (최소 오버헤드)

- 작업 시작 시: 오늘 할 일 3개를 “한 줄”로 적는다.
- 작업 중: 커밋 메시지에 티켓 키(또는 임시 키)를 넣는다.
- 작업 종료 전 5분: “오늘 한 일/내일 할 일/막힌 것”만 3줄로 정리한다.

## 2) 사내 PC에서의 보고 루틴 (복붙 전용)

- 주간보고: 리포지토리에 있는 “Jira 주간보고 텍스트”를 그대로 복붙
- 개발 Tip: 리포지토리에 있는 “Confluence 페이지 초안”을 그대로 복붙 후 사내 링크/표현만 정리

# 리포지토리 내 권장 산출물 구조 (제안)

현재 `docs/`에는 리뷰 문서만 존재하므로, 아래는 “추가 도입 시” 권장안이다.

- `docs/worklog/2026-w05.md`: 주간 작업 로그(SSOT)
- `docs/jira/2026-w05.md`: Jira 주간보고 복붙용 최종 텍스트
- `docs/confluence/tip-<topic>-YYYY-MM-DD.md`: Confluence 개발 Tip 복붙용 초안

사내/대외 민감도 관점에서, 사내 전용 내용이 섞일 위험이 있으면 `docs/internal/` 같은 별도 경로로 분리하고 리포지토리 접근권한을 제한한다.

# Copy & Paste 템플릿

## A) 작업 로그(SSOT) 템플릿: `docs/worklog/2026-wXX.md`

```markdown
# 2026-W05 (2026-01-26 ~ 2026-01-30)

## 이번 주 목표
- (Jira) ABC-123: ...
- (Jira) ABC-456: ...
- (Tip) <topic>: ...

## Daily

### 2026-01-27 (Tue)
- Done:
  - ABC-123: ...
- In progress:
  - ABC-456: ...
- Blockers/Risks:
  - ...
- Notes:
  - PR/Commit: ...
```

## B) Jira 주간보고 템플릿 (복붙용)

Jira 에디터가 무엇이든 깨지지 않도록 “단순 텍스트 + 불릿” 형태로 고정한다.

```text
[주간보고] 2026-W05 (2026-01-26 ~ 2026-01-30)

요약
- 이번 주 핵심 성과 1~2줄

완료 (Done)
- ABC-123: 무엇을/왜/어떤 영향(한 줄)
- ABC-234: ...

진행중 (In Progress)
- ABC-456: 현재 상태 + 남은 작업(한 줄)

다음 주 계획 (Next)
- ABC-789: ...

Blockers / Risks
- (Blocking) ...
- (Risk) ...

참고
- PR: <링크>
- 주요 커밋: <해시 또는 요약>
```

## C) Jira 티켓 설명 템플릿 (사내에서 신규 티켓 생성 시)

```text
배경/문제
- 왜 해야 하는가 (현상/영향)

목표 (Acceptance Criteria)
- Done의 정의를 불릿으로 2~5개

작업 항목
- [ ] ...
- [ ] ...

리스크/의존성
- ...

검증 방법
- 재현/테스트/모니터링
```

## D) Confluence 개발 Tip 템플릿 (복붙용 초안)

```markdown
# <Tip 제목>

## TL;DR
- 한 줄 결론 1~2개

## 문제
- 어떤 상황에서 문제가 발생했는가

## 원인
- 근본 원인(가능하면 “관찰 가능한 증거” 포함)

## 해결
- 해결 방법 요약

## 재현/검증
- 재현 절차
- 검증 절차

## 주의사항
- 같은 실수를 반복하지 않기 위한 체크리스트

## 참고
- 관련 PR/커밋/문서 링크
```

# “make-jira / make-confluence” 자동화(스킬) 제안

사내에서 LLM을 못 쓰더라도, **외부 PC에서** “작업 로그 + git log”를 바탕으로 복붙 산출물을 생성해 커밋해두면 사내에서는 소비만 하면 된다.

## 권장 형태

- 1차(가벼움): 템플릿 기반 수동 작성 (위 템플릿을 그대로 사용)
- 2차(반자동): `docs/worklog/`와 `git log`를 모아 “초안” 파일을 생성하는 스크립트
- 3차(LLM 보조): 스크립트가 뽑은 초안을 LLM이 문장 다듬기/요약/중복 제거 (외부 PC 한정)

## 구현 방향(리포지토리 규칙 준수)

- 실행 스크립트는 `shell-common/tools/custom/`에 둔다. (예: `make_jira.sh`, `make_confluence.sh`)
- CLI로 실행하고 결과는 `docs/jira/`, `docs/confluence/`에 파일로 생성한다.
- LLM 연동은 “옵션”으로 두고, 연동 없이도 동작하도록 만든다.

## 스킬로 묶는 방식(공유 목적)

- `claude/skills/make-jira/SKILL.md`, `claude/skills/make-confluence/SKILL.md` 같은 형태로 “입력 질문 템플릿 + 생성 규칙”을 고정
- 동료는 (외부 PC에서) `skill-loader make-jira`로 스킬 내용을 불러와 Codex/Gemini에도 그대로 적용 가능
