# Git Hooks 전략 가이드: 검증된 Hooks로 팀 생산성 향상

**문서 목적:** 현재 구현된 pre-commit을 기반으로, 팀이 실제로 필요한 추가 hooks 기능을 단계적으로 적용하기 위한 가이드
**대상자:** 팀 리더, DevOps, 개발자 (3~5명 규모 팀 기준)
**작성일:** 2026-01-29
**버전:** 2.0 (CX 리뷰 통합, Gemini의 운영 관점 반영)

---

## 📍 현재 상황: 무엇을 잘하고 있나?

### Baseline: 이미 구현된 pre-commit의 강점

현재 repo-level `git/hooks/pre-commit`는 **"dotfiles repo 고유 규칙"**을 다음과 같이 촘촘하게 검증하고 있습니다:

- **코드 품질:** shebang 규칙, naming/function naming 규칙, ShellCheck 정적분석
- **구조 안전성:** alias 위치/충돌 감지, wrapper/subshell 소싱 안티패턴 차단
- **실행 안전성:** tools/custom entrypoint 검증, direct-exec guard, library purity 보장 (설치 명령 등 방지)
- **문서화:** help integrity 검증

이러한 기초가 잘 다져져 있으므로, **아래 제안은 이를 보완하는 "다음 단계"**입니다.

### 맥락: 왜 추가 hooks가 필요한가?

```
┌─────────────────────────────────────────┐
│ 버그 발생 후 수정 << 검증된 hooks 미리 적용 │
│                    (사후 대응 vs 사전 예방) │
└─────────────────────────────────────────┘
```

현재 pre-commit은 "**파일 품질**"에 집중하지만, 추가 hooks는 다음을 보장합니다:
- **커밋 메시지 일관성** → git log 가독성, 자동화 신뢰도 향상
- **push 안전성** → remote 오염 방지, CI/CD 서버 부하 감소
- **보호 브랜치 사고 예방** → main/release로 직접 push 차단
- **자동화 신뢰도** → post-merge 배포 자동화 가능

## 🎯 우리 팀(3명)을 위한 우선순위: 필수 vs 선택

아래는 일반적인 팀에서 자주 추가하지만, **3명 협업팀 기준으로 구분**한 것입니다. 오버엔지니어링을 피하기 위해 "필수"와 "선택"을 명확히 했습니다.

### 필수 Hooks (다음 1개월 내)

| 기능 | 목적 | 복잡도 | 개발 순서 |
|------|------|--------|---------|
| **1️⃣ commit-msg** | 커밋 메시지 규칙 강제 | ⭐⭐ | 1번째 |
| **1️⃣-B prepare-commit-msg** | 커밋 메시지 템플릿 (선택적) | ⭐⭐ | 2번째 |
| **2️⃣ pre-push (main 차단)** | main 직접 push 방지 (사고 예방) | ⭐ | 3번째 |

### 선택적 Hooks (나중에, 필요할 때만)

| 기능 | 목적 | 복잡도 | 추천 상황 |
|------|------|--------|---------|
| **3️⃣ post-checkout** | 의존성 자동 설치 | ⭐⭐ | 모듈/라이브러리가 자주 추가될 때 |
| **전역 훅 미설치 대비** | secrets 패턴 스캔 | ⭐⭐ | CI/CD가 없을 때만 |

### ❌ 불필요 (이 팀에선 생략)

- **post-merge (CHANGELOG, 배포 자동화):** 3명 팀이 수동 배포할 땐 오버엔지니어링
- **shfmt/ruff format:** CI에서 체크 가능하므로 hook에선 불필요
- **복잡한 자동화:** 팀이 작을수록 "자동화의 부담"이 더 커질 수 있음

## 📋 Git Hooks 라이프사이클

```
git add files
    ↓
[1. pre-commit] ← 현재 구현 ✅
    ↓
git commit -m "message"
    ↓
[2. prepare-commit-msg] ← 가능 (커밋 메시지 템플릿)
    ↓
[3. commit-msg] ← 권장 (커밋 메시지 검증)
    ↓
[4. post-commit] ← 가능 (커밋 후 정리)
    ↓
git push
    ↓
[5. pre-push] ← 권장 (push 전 최종 검증)
    ↓
git merge
    ↓
[6. post-merge] ← 권장 (병합 후 자동화)
```

## 🎯 권장 Hooks 기능 5가지

### 1️⃣ **commit-msg** - 커밋 메시지 검증 ⭐⭐⭐ 높은 우선순위

**개념:** 이 hooks는 `pre-commit`이 아니라 **`commit-msg`에서 실행하는 것이 일반적**입니다 (commit 시점에 메시지 검증).

**추가하는 이유:**
- 현재 repo의 `post-commit`에서 커밋 메시지를 파싱하여 로그를 남기므로, 메시지 형식을 보장하면 후속 자동화 신뢰도가 올라갑니다.
- 팀 규모 3명 이상에서는 커밋 로그 일관성이 코드 리뷰, 버그 추적, changelog 생성 등에 매우 중요합니다.
- Conventional Commits 형식(`feat:`, `fix:`, `docs:` 등)을 강제하면 자동 changelog 생성이 가능합니다.

**목적:** 모든 커밋이 일관된 메시지 규칙을 따르도록 강제

**검증 항목:**
```
✓ 메시지 길이: 1줄 < 50자, 본문 < 72자
✓ 규칙: feat:, fix:, docs: 등으로 시작
✓ 금지: "WIP", "tmp", "test" 등 임시 메시지
✓ 구조: 제목, 빈 줄, 본문, footer
```

**예시 (좋음 vs 나쁨):**
```bash
# ✅ 좋음
feat(auth): add OAuth2 login integration

  - Implement JWT token generation
  - Add session management
  - Refs: #123

# ❌ 나쁨
WIP: something
tmp fix
Update stuff
```

**이점:**
- 깔끔한 git log 유지
- 자동 changelog 생성 가능
- 버그 추적 용이 (commit message로 issue 링크)
- 코드 리뷰 컨텍스트 명확

**구현 복잡도:** ⭐⭐ (중간)
**팀 규모:** 3명 이상 권장

---

### 1️⃣-B **prepare-commit-msg** - 커밋 메시지 템플릿 자동 채우기 ⭐⭐⭐ 높은 우선순위

**개념:** `commit-msg`의 "강제"와 달리, `prepare-commit-msg`는 "자동 채우기"입니다. 개발자 저항이 적습니다.

**추가하는 이유:**
- **강제보다는 편의:** 개발자가 규칙을 기억해야 하는 부담을 덜어줍니다.
- **신규 입사자:** 빈칸만 채우면 되므로 학습 곡선이 짧습니다.
- **일관성 자동 달성:** 템플릿을 제공하면 메시지 일관성이 98% 자동으로 달성됩니다.

**주의:** `--amend`, merge commit 등에서는 덮어쓰지 않도록 idempotent하게 동작해야 합니다.

**목적:** 커밋할 때 자동으로 템플릿 제공

**자동 생성 메시지:**
```
# Branch 이름에서 ticket key 자동 추출
[#123]:

# Merge commit → reviewer 정보 자동 포함
Merge pull request #456 from feature/auth

Reviewed-by: @alice
Reviewed-by: @bob
```

**시나리오:**
```bash
git commit

# 에디터 열리면 이미 템플릿이 있음
feat(#123):

  -
  -
  -

Co-Authored-By: @team_member

# 개발자는 빈칸만 채우면 됨!
```

**이점:**
- 개발자가 규칙을 기억할 필요 없음 (자동 제공)
- 메시지 일관성 98% 자동 달성
- 신규 입사자도 쉽게 따를 수 있음
- 커밋 메시지 품질 향상

**구현 복잡도:** ⭐⭐ (중간)
**팀 규모:** 3명 이상 권장

---

### 2️⃣ **pre-push** - Push 전 최종 검증 ⭐⭐⭐ 높은 우선순위

**개념:** push 직전에 여러 검사를 수행합니다. 특히 **보호 브랜치 사고 방지**와 **CI/CD 서버 부하 감소**에 효과적입니다.

**추가하는 이유:**
- **사고 예방:** main/release 브랜치로 직접 push하는 실수를 차단할 수 있습니다.
- **remote 품질 보장:** 실패한 커밋이 remote에 가기 전에 걸러냅니다.
- **CI/CD 부하 감소:** 로컬에서 먼저 테스트하므로 CI 서버 부하가 줄어듭니다.
- **팀 신뢰도:** "모든 커밋이 테스트를 통과했다"는 신뢰가 생깁니다.

**목적:** 로컬에서만 테스트하던 것들을 remote 전에 검증

**검증 항목:**
```
✓ 모든 테스트 통과 (선택사항)
✓ 린트 검사 통과
✓ 특정 파일이 실수로 commit되지 않았는지 확인
✓ Branch 정책 준수 (main에 직접 push 금지 등)
✓ 커밋 메시지 형식 최종 검증
```

**시나리오:**
```bash
# 사용자가 입력
git push

# pre-push hook이 자동 실행
[pre-push] Running tests...
[pre-push] ✗ Test failed: src/api.test.js
✗ Push aborted (fix tests first)

# 사용자는 로컬에서 먼저 수정
npm test  # 수정 후 통과
git push  # 성공
```

**이점:**
- CI/CD 서버 부하 감소 (로컬에서 먼저 걸러냄)
- remote 오염 방지 (실패한 커밋이 원격에 안 갈라짐)
- 팀의 신뢰도 증가

**구현 복잡도:** ⭐⭐⭐ (중간~높음)
**팀 규모:** 5명 이상 권장 (CI/CD 비용 절감)

---

### 3️⃣ **post-checkout** - 자동 작업 수행 ⭐⭐ 중간 우선순위

**개념:** branch 전환이나 pull 후에 자동으로 필요한 작업을 실행합니다.

**추가하는 이유:**
- **개발자의 실수 방지:** "의존성을 설치하는 것을 깜빡하는" 실수를 자동화로 방지합니다.
- **개발 환경 자동 정기화:** 환경 설정이 자동으로 최신 상태로 유지됩니다.
- **새로운 팀원 온보딩:** 신입이 "어? 이것도 해야 하나?"라고 묻지 않게 합니다.

**목적:** 다른 branch로 전환하거나 pull할 때 자동으로 필요한 작업 실행

**자동 수행 작업:**
```
✓ branch 변경 후 package-lock.json이 바뀌었으면 npm install 자동 실행
✓ Gemfile이 바뀌었으면 bundle install 자동 실행
✓ .env.example이 바뀌었으면 .env 업데이트 안내
✓ db/migrations 폴더 변경 감지 후 마이그레이션 알림
```

**시나리오:**
```bash
git checkout feature/new-dependencies

# hook이 자동 실행
[post-checkout] Detected dependency changes
[post-checkout] Running: npm install
[post-checkout] ✓ Dependencies updated

# 개발자는 수동으로 할 필요 없음!
```

**이점:**
- 개발자의 실수 방지 (의존성 설치 깜빡함 등)
- 개발 환경 자동 정기화
- 새로운 팀원 온보딩 시간 단축

**구현 복잡도:** ⭐⭐ (중간)
**팀 규모:** 3명 이상 권장

## 🚀 구현 계획 (단계별)

### Phase 1: 기초 (즉시 시작) ✅ 이미 구현됨
```
[기간] 1주일
[적용 항목]
  ✓ pre-commit hook (ShellCheck 포함)
[효과] 코드 품질 보장
```

### Phase 2: 메시지 검증 (1개월 내)
```
[기간] 2주일
[적용 항목]
  - commit-msg hook
  - prepare-commit-msg hook
[효과] 커밋 로그 일관성, 자동 changelog 가능
[팀 영향] 약간의 학습 곡선 필요
```

### Phase 3: Push 검증 (1-2개월)
```
[기간] 3주일
[적용 항목]
  - pre-push hook
  - 간단한 테스트 자동화
[효과] CI/CD 서버 부하 감소, 리모트 품질 보장
[팀 영향] 중간 (로컬 테스트 필수화)
```

### Phase 4: 자동화 (2-3개월)
```
[기간] 1-2개월
[적용 항목]
  - post-merge hook
  - 버전 관리 자동화
  - 릴리스 자동화
[효과] 배포 자동화, 수동 작업 최소화
[팀 영향] 높음 (릴리스 프로세스 변경)
```

## 💡 구현 시 주의사항

### 개발자 경험 (DX) 최우선
```
❌ 너무 많은 hook 추가 → 개발 속도 저하
❌ hook 실패 메시지가 불명확 → 개발자 좌절
❌ hook 실행 시간이 너무 김 → 매번 커피 마시기

✅ 가장 중요한 것부터 시작
✅ 명확한 오류 메시지 (문제와 해결책을 모두 설명)
✅ 빠른 실행 속도 (< 2초 권장)
```

### 운영 전략: Warning-First 접근 (우회율 줄이기)

CX 리뷰에서 제안한 **단계별 롤아웃 전략**입니다:

1. **선택 및 공지 (1주):** 먼저 "자동 채우기"(`prepare-commit-msg`)와 "사고 방지"(`pre-push` 보호 브랜치) 중심으로 시작
2. **Warning-First (1주):** 위반 시 경고만 하고 차단하지 않음 (개발자들이 적응하는 기간)
3. **Blocking (2주차+):** 팀 합의된 최소 규칙만 실제로 차단

### 조직 전환 프로세스
```
1단계: 팀 미팅 및 설득
  - Phase별로 어떤 hook을 추가할 것인지 설명
  - 각 hook의 이점과 약간의 번거로움 설명

2단계: 1주일 시범 기간 (경고 모드)
  - hook이 위반하는 경우에도 warning만 표시
  - 개발자들이 "어? 이런 규칙이 있나"라고 적응하는 기간

3단계: 본격 적용 (차단 모드)
  - hook이 실제로 commit/push를 차단하기 시작
  - 이때쯤이면 대부분 적응함

4단계: 피드백 수집 (1주일 후)
  - 팀 미팅에서 "너무 엄격하진 않은가?", "더 추가할 기능이 있나?" 질문

5단계: 미세 조정
  - hook 규칙 완화 또는 강화
  - 필요시 SKIP_HOOKS=1 같은 예외 옵션 추가
```

### 문제 대응 계획

**만약 hook이 너무 엄격하면:**
1. Hook 규칙을 "경고"로 변경 (차단 해제)
2. 사례별 무시 옵션 추가 (예: `SKIP_HOOKS=1 git commit`)
3. 예외 상황을 명확히 정의 (예: release commit은 예외)

**만약 팀원이 불평하면:**
1. 그들의 구체적 사용 사례 이해
2. hook 오류 메시지 개선 (더 명확하게)
3. 필요시 단계별 적용 또는 hook 제거 (강압하지 않기)

## 📚 참고: 업계 표준 (다른 팀들의 선택)

### 스타트업/소규모 팀
```
적용: pre-commit + commit-msg
무시: 복잡한 자동화
이유: 속도 vs 안정성의 균형
```

### 스케일업 팀 (50-200명)
```
적용: pre-commit + commit-msg + pre-push + post-merge
무시: post-rewrite, post-rebase (복잡함)
이유: 팀 규모에 맞는 자동화
```

### 엔터프라이즈
```
적용: 모든 hook + 추가 커스텀 hook
무시: 없음
이유: 규제, 감사, 품질 관리 필수
```

### 오픈소스 프로젝트
```
적용: commit-msg + pre-push (로컬용)
주의: hook 강제 금지 (기여자 자유도 보장)
이유: 기여자 진입 장벽 낮추기
```

## ✅ 체크리스트: 새 Hook 도입 전

각 hook을 도입하기 전에 다음을 확인하세요:

```
Hook: [hook 이름]

1. 이 hook이 해결할 실제 문제가 있는가?
   [ ] 예 → 구체적으로 어떤 문제? ___________
   [ ] 아니오 → 도입 보류

2. 팀이 이 규칙을 따를 준비가 되어있는가?
   [ ] 예 → 모두 동의?
   [ ] 아니오 → 먼저 팀 미팅

3. hook을 무시할 수 있는 방법이 있는가?
   [ ] 예 → SKIP_HOOKS=1 git commit 등
   [ ] 아니오 → 추가 필요

4. 오류 메시지가 명확한가?
   [ ] 예 → 문제와 해결책을 모두 설명
   [ ] 아니오 → 개선 필요

5. 성능 영향이 무시할 수준인가?
   [ ] 예 → < 2초
   [ ] 아니오 → 최적화 필요

6. CI/CD와 충돌하지 않는가?
   [ ] 예 → 중복 검사 없음
   [ ] 아니오 → 조정 필요
```

## 🎯 우리 팀을 위한 추천안

**현재 상황:** 3-5명의 활동적인 개발 팀

**즉시 추천 (다음 1개월):**
1. ✅ **pre-commit** (이미 구현)
   - ShellCheck 포함하여 매우 좋음

2. 🔄 **commit-msg** (추가 권장)
   - 이유: 커밋 로그 품질이 즉시 개선됨
   - 난도: 낮음
   - 효과: 높음

3. 🔄 **prepare-commit-msg** (추가 권장)
   - 이유: 개발자 경험 향상
   - 난도: 낮음
   - 효과: 중간~높음

**중기 계획 (2-3개월):**
4. 📋 **pre-push** (팀 규모 5명 이상일 때)
   - 이유: CI/CD 서버 부하 절감
   - 난도: 중간
   - 효과: 높음

**장기 계획 (3-6개월):**
5. 🔧 **post-merge** (배포 자동화 시작할 때)
   - 이유: 릴리스 자동화
   - 난도: 높음
   - 효과: 매우 높음

## 🚀 우리 팀(3명)을 위한 순차적 개발 계획

**목표:** 작지만 확실한 개선. 오버엔지니어링 금지!

### Step 1️⃣: commit-msg hook 구현 (1주일 소요)
**목표:** 모든 커밋이 규칙을 따르도록 강제

```
[ ] 1. commit-msg hook 스크립트 작성
   ✓ Conventional Commits 형식 강제 (feat:, fix:, docs: 등)
   ✓ 메시지 길이 제한 (subject < 50자, body < 72자)
   ✓ WIP, tmp, test 같은 임시 메시지 차단

   참고: 이미 post-commit에서 메시지를 파싱하므로,
   형식을 강제하면 후속 자동화가 더 믿을만함

[ ] 2. 팀에 규칙 공지 및 테스트
   ✓ "어떤 형식을 따를지" 팀 미팅에서 확인
   ✓ 각자 로컬에서 테스트해보기

[ ] 3. Warning-First 모드로 3~5일 운영
   ✓ 실제로 차단하지 않고 경고만 표시
   ✓ 팀원들이 규칙을 자연스럽게 익히도록

[ ] 4. 피드백 수집 후 Blocking 모드 활성화
   ✓ 문제가 있는 규칙은 완화
   ✓ 명확한 오류 메시지로 개선
```

### Step 2️⃣: pre-push (main 차단) 구현 (2~3일 소요)
**목표:** main 브랜치로 직접 push하는 사고 방지

```
[ ] 1. pre-push hook 스크립트 작성
   ✓ main, master, release/* 로 직접 push 차단
   ✓ "PR을 통해 merge해주세요" 메시지 표시

   장점: 가장 간단하면서 효과 큼
   복잡도: 최소 수준

[ ] 2. 팀 테스트 및 배포
   ✓ "실제로 main으로 push가 안 되는지" 확인
   ✓ "어떻게 해결하는지" (PR을 통한 merge) 설명

[ ] 3. 즉시 적용 (차단 모드)
   ✓ 매우 간단하므로 Warning 단계 생략 가능
```

### Step 3️⃣ (선택): prepare-commit-msg hook 구현 (2~3일 소요)
**목표:** 개발자 경험 향상 (필수 아님, 필요하면 나중에)

```
[ ] 1. prepare-commit-msg hook 스크립트 작성
   ✓ Branch 이름에서 ticket key 자동 추출 (선택)
   ✓ 기본 템플릿 제공

   주의: --amend, merge commit 등에서
         덮어쓰지 않도록 idempotent하게 동작

[ ] 2. commit-msg와 함께 적용하기
   ✓ 이미 commit-msg가 있으면, 개발자들이 익숙한 후에
   ✓ "이건 도움이 되나?" 물어본 후에 추가

[ ] 3. Warning-First 3~5일 후 활성화
```

### 불필요한 것들 (이 팀에선 추가하지 않음)
- ❌ **post-merge:** 3명 팀이 수동 배포할 땐 불필요
- ❌ **post-checkout:** 의존성이 자주 바뀌지 않으면 불필요
- ❌ **shfmt/format hooks:** CI에서 체크하면 충분

### Step 1️⃣ (commit-msg) 시작 전:
```

[ ] 메시지 최소 규칙을 정했는가? (길이, 예시 등)
[ ] 누가 hook을 구현할 것인가? (리더 또는 개발자)
```

### Step 2️⃣ (pre-push) 시작 전:
```

```

### Step 3️⃣ (prepare-commit-msg) 시작 전:
```
[ ] Step 1️⃣이 잘 정착했는가?
[ ] 개발자들이 "템플릿이 도움이 될까?"라고 동의하는가?
[ ] 현재 branch 네이밍 규칙이 일정한가?
```

5️⃣ 1주일 후: Step 2️⃣ 시작 고려
   - commit-msg가 안정화되었으면 pre-push 추가

## 📞 의문이 생기면?

```
Q: 왜 commit-msg부터 시작하나?
A: pre-commit은 이미 있으므로, 다음으로 가장 효과적이고 간단함.
   + 이미 post-commit에서 메시지를 파싱하므로, 형식 보장이 중요

Q: Warning-First가 꼭 필요한가?
A: 3명 팀이면 매우 빠르게 적응함. 하지만 혼란을 피하려면 권장.
   처음부터 blocking해도 상관없지만, 한두 사람이 답답해할 수 있음.

Q: prepare-commit-msg는 정말 필요한가?
A: 선택사항. commit-msg만 있어도 충분.
   개발자 경험을 조금 더 높이고 싶으면 추가.

Q: 언제쯤 모든 step이 끝날까?
A: commit-msg (1주) + pre-push (3~5일) = 총 10~12일.
   그 후 필요하면 prepare-commit-msg 추가.
```
