# JIRA Agent (Phase 1) - CX/Use Case 요구사항

## 1. 배경/문제 정의
- 새로운 팀(약 240명)의 업무 관리 그라운드룰이 **JIRA 기반**으로 정해짐
- 다수의 개발자가 “JIRA 티켓 생성”을 **귀찮고 불편한 작업**으로 인식
- 목표: 사용자가 텍스트(회의록/긴급 업무 메모 등)만 입력하면, LLM이 내용을 요약해 **Action Item(A/I) 2~3개 중심의 JIRA Task**를 빠르게 생성(필요 시 sub-task까지)

## 2. Phase 1 목표(Outcomes)
- 입력 텍스트에서 핵심 맥락을 파악해 **JIRA 생성에 필요한 최소 정보**를 제안
- 사용자는 “AI가 만든 초안”을 **검토/수정**한 뒤 JIRA에 생성
- 결과물(티켓)은 팀 규칙(명확한 제목, 실행 가능한 A/I, 담당/기한/우선순위 등)에 맞게 **일관된 품질**로 생성

## 3. Phase 1 범위(In/Out)

### In Scope (Phase 1)
- 웹 기반 Frontend에서 텍스트 입력 → 요약/정리 → JIRA Task 생성
- 회의록/채팅로그/긴급 업무 메모 등 다양한 입력 포맷 지원(붙여넣기 중심)
- A/I 2~3개를 중심으로 Task 생성(기본)
- 옵션: sub-task 자동 생성(사용자 선택 또는 규칙 기반 추천)
- 생성 전 Preview 제공(제목/설명/A/I/필드 편집 가능)
- 기본적인 실패 처리(권한/네트워크/필드 검증 오류 등) 및 재시도 UX

### Out of Scope (Phase 1, Phase 2 후보)
- Epic/Story를 읽고 자동 연결(상위 이슈 연결 자동화)
- JIRA comment 업데이트 기반 메일 알림
- Confluence MCP 연동(추가 서비스/지식 연결)

## 4. 주요 사용자(페르소나)
- **개발자(Contributor)**: 회의에서 정해진 일을 티켓화하는 시간이 아까움. 빠른 생성 + 최소 수정 원함.
- **Tech Lead/PM(Organizer)**: 회의록에서 “누가/무엇을/언제까지”를 명확히 티켓으로 남기고 추적하고 싶음.
- **온콜/긴급 대응 담당자(Responder)**: 긴급 업무를 빠르게 티켓화하고 우선순위/담당/기한을 명확히 남기고 싶음.

## 5. 기본 UX 원칙(Phase 1)
- **입력 → 제안 → 검토/편집 → 생성** 4단계로 인지부하 최소화
- “자동 생성”은 초안이며, **최종 책임은 사용자 검토**에 있음(Preview 필수)
- 편집은 “JIRA 폼을 그대로 복제”하기보다, 사용자가 자주 고치는 필드 위주로 제공
- 생성 실패 시 **원본 입력/AI 초안이 사라지지 않음**(재시도/복사/저장)

## 6. 핵심 사용자 여정(Happy Path)
1. 사용자가 웹 화면에서 회의록/메모를 붙여넣음
2. (선택) 프로젝트/이슈 타입(Task)/우선순위/라벨 등 최소 메타를 선택
3. “요약/티켓 초안 생성” 클릭
4. 시스템이 A/I 2~3개 및 티켓 초안을 생성(제목, 설명, 체크리스트/서브태스크 제안)
5. 사용자는 Preview에서 문구/담당자/기한 등을 수정
6. “JIRA 생성” 클릭 → 생성된 이슈 키/링크 반환

## 7. 입력 유형별 상세 Use Case

### UC-01: 회의록 → 단일 Task 생성(A/I 2~3개)
**상황**: 스프린트/설계/리뷰 회의가 끝났고 회의록이 있음.  
**입력 예시**
```text
- 로그인 실패 케이스가 간헐적으로 발생. 원인 후보: 토큰 갱신 경합
- 다음 주 배포 전까지 재현 시나리오 작성 필요
- API rate limit 문서가 outdated. 정리 필요
결정:
- 토큰 갱신 로직에 락 추가 검토
Action:
- (민수) 재현 스크립트 작성
- (지은) 문서 업데이트
```
**시스템 동작(요구)**
- 제목: 핵심 작업을 한 문장으로(명사형/동사형 혼용 가능하되 간결)
- 설명: 배경/맥락(왜 필요한지), 결정 사항, 참고 링크(있다면)
- A/I: 2~3개로 압축(가능하면 “동사+대상+산출물” 형태)
- (선택) 담당자/기한 후보를 추출해 제안(확정은 사용자가)
**대체 흐름**
- 회의록이 너무 짧거나 모호 → 질문 형태의 “확인 필요 사항” 1~2개를 제안하되, 그래도 초안을 생성
**성공 기준**
- 사용자가 1분 내 Preview에서 약간 수정 후 생성 가능

### UC-02: 회의록 → Task + sub-task 세분화 생성
**상황**: 회의에서 나온 작업이 여러 단계로 나뉘며 담당도 다름.  
**시스템 동작(요구)**
- 상위 Task: 목표/완료 정의(Definition of Done)를 포함
- sub-task: 2~5개 범위로 제안(너무 잘게 쪼개지지 않게)
- sub-task에 각각 명확한 완료 조건(검증 방법/산출물)을 포함
**UX 요구**
- “sub-task로 쪼개기” 토글 + 생성 전 리스트 편집(추가/삭제/순서 변경)
**성공 기준**
- 사용자가 sub-task를 일부 삭제/병합하고도 빠르게 생성 가능

### UC-03: 긴급 업무 메모(온콜/장애/핫픽스) → Task 생성(A/I 2~3개)
**상황**: “긴급 업무”를 전화/메신저로 받았고, 정리되지 않은 메모만 있음.  
**입력 예시**
```text
프로덕션에서 결제 승인 콜백이 5xx 증가. 09:40부터. 원인 불명.
우선 임시로 재시도 횟수 상향 + 알람 임계값 조정 필요.
추가로 로그 샘플 수집하고 재발 방지책 정리해야 함.
```
**시스템 동작(요구)**
- 우선순위/심각도(있다면) 추출해 제안(예: High)
- A/I 2~3개에 “즉시 조치”와 “원인 분석/재발 방지”가 균형 있게 포함
- 타임라인/영향 범위가 있으면 Description 상단에 요약
**대체 흐름**
- 정보가 부족한 경우: “필수 확인 항목(서비스/지표/에러 로그/발생 시각)”을 체크리스트로 제안

### UC-04: 여러 안건이 섞인 회의록 → 이슈 2~N개로 분리 생성(선택)
**상황**: 하나의 회의록에 서로 다른 주제(문서, 버그, 성능, 운영)가 섞여 있음.  
**시스템 동작(요구)**
- 기본은 “대표 Task 1개” + A/I 2~3개
- 옵션으로 “여러 이슈로 나누기”를 제공:
  - 주제/컴포넌트별로 2~N개의 초안 생성
  - 각 초안에 해당 A/I만 포함(중복 최소화)
**UX 요구**
- 생성 전: 분리된 초안을 탭/리스트로 보여주고, 일부만 선택 생성 가능

### UC-05: 채팅 로그(슬랙/팀즈) → 결정/액션만 추출해 Task 생성
**상황**: 비동기 논의가 길게 이어졌고, 마지막에 결정과 액션이 있음.  
**시스템 동작(요구)**
- 대화의 잡음을 줄이고 “결정(Decision)”과 “액션(Action)”만 남김
- 누가 말했는지(이름/아이디)에서 담당 후보를 추출해 제안(확정은 사용자)
**대체 흐름**
- 결론이 없는 토론 로그 → “결정 필요 사항” 1~2개를 상단에 명시하고, 액션은 최소화

### UC-06: 개인 작업 메모(불릿/체크리스트) → Task로 정돈
**상황**: 사용자가 TODO를 대충 적어두었고, 티켓화가 귀찮음.  
**시스템 동작(요구)**
- 중복/유사 항목을 병합하여 A/I 2~3개로 재구성
- 너무 큰 항목은 sub-task 제안(옵션)

### UC-07: “티켓 제목만 대충” 입력 → 질문/보완을 통해 생성
**상황**: 입력이 “로그인 버그 처리” 정도로 지나치게 짧음.  
**시스템 동작(요구)**
- 초안은 생성하되, Preview에서 “질문(필수/권장)”을 표시:
  - 필수: 영향 범위, 재현 방법(가능하면), 완료 기준
  - 권장: 우선순위, 담당, 기한
- 사용자가 답을 채우면 Description/A/I에 자동 반영

### UC-08: 사용자가 A/I 개수를 조정(2~3개 기본, 필요 시 변경)
**상황**: 팀 규칙은 2~3개가 이상적이나, 예외적으로 더 필요함.  
**시스템 동작(요구)**
- 기본: 2~3개
- 사용자가 “A/I 더 보기”를 누르면:
  - 4~6개 후보를 생성하되, 중복/의존성을 함께 표시
  - 사용자가 최종 포함 항목을 선택하도록 UX 제공

### UC-09: 기존 JIRA 이슈에 sub-task만 추가 생성
**상황**: 상위 Task는 이미 존재하고, 회의/메모에서 나온 실행 항목만 sub-task로 추가하고 싶음.  
**시스템 동작(요구)**
- 사용자가 Parent 이슈 키(예: `PROJ-123`)를 입력/선택
- 입력 텍스트에서 A/I를 추출해 sub-task 2~5개 초안 생성
- Preview에서 sub-task만 편집/선택 생성 가능(상위 Task 수정 없이)
**대체 흐름**
- Parent 키가 잘못되었거나 권한이 없음 → 검증 오류를 즉시 표시하고 초안은 유지

### UC-10: 민감정보/개인정보 포함 입력 → 경고 및 마스킹 제안
**상황**: 회의록/메모에 전화번호, 계정정보, 고객식별자 등 민감정보가 섞일 수 있음.  
**시스템 동작(요구)**
- 생성 전 Preview에서 “민감정보로 의심되는 문자열”을 하이라이트 및 경고
- (선택) 자동 마스킹 제안(사용자가 적용/되돌리기 가능)
- JIRA로 전송되는 최종 내용은 Preview의 편집 결과를 그대로 사용

## 8. JIRA 생성 전 Preview/편집 요구사항
- 필수 편집 필드(권장):
  - Project, Issue Type(Task), Summary(제목), Description(본문)
  - A/I(체크리스트 또는 Description 내 섹션)
- 선택 편집 필드(팀 설정에 따라):
  - Assignee(담당), Due date(기한), Priority, Labels, Components, Sprint/Board 필드(가능할 경우)
- sub-task가 있을 때:
  - 각 sub-task Summary/Description 편집
  - 담당/기한은 상속 옵션 제공(상위 Task 값 상속 후 개별 수정)

## 9. 오류/예외 UX(Phase 1)
- JIRA 권한/인증 실패:
  - 로그인/토큰 갱신 안내 + 현재 초안 유지
- 필수 필드 누락/검증 오류:
  - 어떤 필드가 왜 실패했는지 구체적으로 표시 + 해당 필드로 포커스 이동
- JIRA API 실패/타임아웃:
  - 재시도 버튼 + 초안 복사(클립보드) + 로컬 임시 저장(가능하면)
- LLM 응답 실패:
  - “간단 모드(템플릿 기반)”로 폴백(사용자 입력에서 제목/본문만으로 최소 티켓 생성)

## 10. 품질 기준(Phase 1)
- 생성되는 Task는 최소한 아래를 만족:
  - 제목이 모호하지 않음(대상/행동이 드러남)
  - Description에 배경/목표/완료 조건 중 2개 이상 포함
  - A/I가 실행 가능하고 측정 가능(산출물/검증 기준 포함)

## 11. 예시 출력 포맷(가이드)
```markdown
## Context
- 배경/현상:
- 결정 사항:

## Action Items
- [ ] (A/I1) ...
- [ ] (A/I2) ...
- [ ] (A/I3) ...

## Definition of Done
- ...
```

## 12. Phase 2 아이디어(참고)
- Epic/Story를 읽고 Task 생성 시 자동 연결
- JIRA comment 업데이트 시 assigner에게 메일 noti
- Confluence MCP 연동(회의록/문서와 티켓 연결 강화)
