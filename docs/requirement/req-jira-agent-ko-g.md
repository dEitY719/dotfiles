# JIRA Agent - 요구사항 명세서 (Phase 1)

## 1. 개요
JIRA Agent는 AI(LLM)를 활용하여 JIRA 티켓 생성 과정을 간소화하는 서비스입니다. 회의록, 미정리 노트, 긴급 메시지 등 비정형 데이터를 분석하여 구조화된 JIRA Task와 Sub-task로 변환합니다. 이를 통해 팀의 JIRA 운영 부담을 줄이고 개발 생산성을 높이는 것을 목표로 합니다.

## 2. 목표
- 자연어 입력을 통한 JIRA 티켓 생성 자동화
- AI 처리를 통한 고품질 티켓 메타데이터(요약, 설명, 담당자 등) 확보
- 생성 전 사용자가 내용을 검토하고 수정할 수 있는 웹 인터페이스 제공

## 3. 핵심 기능 (Phase 1)
- **AI 작업 추출**: 긴 텍스트에서 2~3개의 핵심 Action Item을 추출하여 JIRA Task 제안
- **계층 구조 분해**: 복잡한 요구사항을 자동으로 Sub-task로 세분화
- **웹 인터페이스**: 텍스트 입력, AI 제안 내용 확인 및 수정, 최종 생성 확인 UI
- **JIRA 연동**: JIRA API(OAuth 또는 PAT)를 통한 보안 연결 및 티켓 생성
- **스마트 매핑**: 입력 문맥에 따른 프로젝트, 컴포넌트, 우선순위 자동 설정

## 4. 상세 Use Case

### UC 1: 회의록 기반 액션 아이템 생성
- **사용자 스토리**: 프로젝트 매니저로서 회의록을 에이전트에 붙여넣어, 논의된 각 작업들을 수동으로 만들지 않고 자동으로 생성하고 싶다.
- **시나리오**:
    1. 사용자가 줌(Zoom) 전사 기록이나 회의 노트를 입력한다.
    2. AI가 주요 결정 사항과 할당된 작업(예: "Jane이 금요일까지 API 문서를 업데이트함")을 식별한다.
    3. AI가 JIRA Task를 제안한다:
        - 제목: API 문서 업데이트
        - 담당자: Jane
        - 마감일: 이번 주 금요일
        - 설명: 회의록에서 추출된 관련 컨텍스트 포함
    4. 사용자가 웹 UI에서 검토 후 "전체 생성"을 클릭한다.

### UC 2: 긴급 Slack 메시지의 티켓화
- **사용자 스토리**: 리드 개발자로서 Slack의 긴급 요청을 즉시 JIRA 작업으로 전환하여 누락 없이 관리하고 싶다.
- **시나리오**:
    1. 사용자가 "결제 게이트웨이에서 500 에러 발생 중. 로그 확인 및 긴급 수정 필요" 메시지를 복사해 넣는다.
    2. AI가 시급성과 기술 영역을 판단한다.
    3. AI가 제안한다:
        - 제목: [긴급] 결제 게이트웨이 500 오류 수정
        - 우선순위: 최고(Highest)
        - 레이블: bug, production, payment
    4. 사용자가 확인하면 즉시 티켓이 생성된다.

### UC 3: 복합 기능 구현의 Sub-task 분해
- **사용자 스토리**: 개발자로서 새로운 기능 구현 요청을 자동으로 하위 작업으로 나누어 체계적으로 관리하고 싶다.
- **시나리오**:
    1. 사용자가 "이미지 업로드 및 비밀번호 재설정 기능이 포함된 사용자 프로필 페이지 구현"을 입력한다.
    2. AI가 여러 컴포넌트를 인식한다.
    3. AI가 제안한다:
        - 상위 Task: 사용자 프로필 페이지 구현
        - Sub-task 1: UI 레이아웃 설계 및 구현
        - Sub-task 2: 이미지 업로드 백엔드 로직 구현
        - Sub-task 3: 비밀번호 재설정 플로우 구현
    4. 사용자가 필요한 하위 작업을 선택하여 계층 구조를 생성한다.

### UC 4: 대화형 검토 및 수정
- **사용자 스토리**: 사용자로서 AI가 제안한 내용 중 미흡한 부분을 직접 수정하여 정확한 티켓을 만들고 싶다.
- **시나리오**:
    1. AI가 생성한 제목이 너무 일반적인 경우, 웹 UI의 텍스트 필드에서 제목을 수정한다.
    2. 드롭다운을 통해 담당자를 변경한다.
    3. AI가 놓친 레이블을 수동으로 추가한다.
    4. 최종 수정된 내용을 JIRA에 제출한다.

### UC 5: 프로젝트 및 컴포넌트 자동 탐지
- **사용자 스토리**: 사용자로서 매번 프로젝트를 선택하지 않아도 에이전트가 문맥을 파악해 적절한 프로젝트를 골라주길 원한다.
- **시나리오**:
    1. 사용자가 "모바일 앱의 메모리 누수 수정"을 입력한다.
    2. AI가 "모바일 앱"을 JIRA의 "MOBILE" 프로젝트와 매칭한다.
    3. 컴포넌트를 "Engine/Core"로 설정한다.
    4. 사용자는 UI에 미리 채워진 정보를 확인한다.

### UC 6: 스마트 중복 감지 (Phase 1 초기 도입)
- **사용자 스토리**: 개발자로서 중복된 작업이 생성되는 것을 방지하기 위해 유사한 티켓이 있는지 확인하고 싶다.
- **시나리오**:
    1. 사용자가 버그 설명을 입력한다.
    2. AI가 최근 생성된 JIRA 티켓들을 검색한다.
    3. AI가 경고한다: "유사한 티켓(BUG-123: 대시보드 로딩 저하)이 이미 존재합니다. 새로 만드시겠습니까?"
    4. 사용자는 새 티켓 생성 대신 기존 티켓을 업데이트하기로 결정한다.

### UC 7: 보안/법적 이슈 우선순위 자동 격상
- **사용자 스토리**: 보안 담당자로서 고위험 이슈가 입력되면 에이전트가 이를 즉시 강조해주길 원한다.
- **시나리오**:
    1. "인증되지 않은 API 엔드포인트에서 사용자 이메일이 유출됨" 입력.
    2. AI가 "인증되지 않은", "유출", "이메일" 키워드를 감지한다.
    3. 우선순위를 즉시 "Blocker"로 설정하고 "security-vulnerability" 레이블을 추가한다.

### UC 8: 다수 작업자 대상 티켓 복제 생성
- **사용자 스토리**: 팀장으로서 동일한 교육이나 공통 작업을 여러 팀원에게 한 번에 할당하고 싶다.
- **시나리오**:
    1. "모든 개발자는 필수 보안 교육을 이수해야 함" 입력.
    2. AI가 이를 그룹 공통 작업으로 인식한다.
    3. UI에서 5명의 개발자를 선택한다.
    4. 에이전트가 각 담당자별로 동일한 내용의 JIRA 티켓 5개를 생성한다.

## 5. 기술 요구사항 (간략)
- **Backend**: Python (FastAPI 권장)
- **AI**: OpenAI GPT-4o 또는 Claude 3.5 Sonnet
- **Frontend**: React 또는 Dash Mantine Components
- **인증**: 초기 MVP는 JIRA PAT(Personal Access Token) 사용, 이후 OAuth 확장

## 6. Phase 1 제약 사항
- Task와 Sub-task 생성에 집중 (Epic, Story는 Phase 2 이후)
- 단방향 생성 (에이전트 -> JIRA)만 지원, JIRA에서의 수정사항 역동기화 제외
- 사전에 구성된 한정된 수의 JIRA 프로젝트만 지원
